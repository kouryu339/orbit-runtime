# 9 上下文结构和快照机制

本文只说明模型实际收到的上下文顺序、尾部快照，以及摘要的位置。这里的“快照”仅指
`conversation.set_dynamic_snapshot` 写入并在每轮 thinking 时注入模型的动态上下文；
它不是 conversation 存档，也不是 Runtime 状态导出。

## 9.1 上下文顺序

每轮调用 LLM 时，消息按以下顺序组成：

```text
1. system message
   - system Skills
   - role Skill
   - 当前激活的 feature Skills
   - 当前可见的工具与 workflow 描述

2. conversation history
   - 历史用户、助手和工具消息
   - 如果存在摘要，摘要位于这里
   - 当前用户消息也属于这一段历史

3. final dynamic-context user message
   - 当前 Agent 的 host dynamic snapshot
   - recorder chain、current plan、retrieval、env/runtime 等本轮动态信息
```

因此，固定关系是：

```text
system -> history (including summary) -> final dynamic context (including snapshot)
```

不要通过 cluster 额外注入未定义的固定上下文字段。稳定职责、行为规则和能力边界应写入
role Skill；按任务变化的工具能力应写入 feature Skill 并通过 `UpdateSkills` 激活。

## 9.2 尾部快照

宿主通过 `conversation.set_dynamic_snapshot` 为指定 Agent 发布当前业务上下文：

```json
{
  "schema": "agent-runtime-command/v1",
  "type": "conversation.set_dynamic_snapshot",
  "payload": {
    "conversation_id": "conv-current",
    "agent_id": "service-agent-1",
    "field_name": "current_order",
    "text": "Order SO-1008 is awaiting payment."
  }
}
```

Runtime 按 `(conversation_id, agent_id, field_name)` 保存最新文本；再次写入同一个
`field_name` 会覆盖旧值。一个 Agent 可以有多个字段，但它们不是多份模型快照：Runtime
会按字段名稳定排序、合并字段文本，并在每轮 thinking 时把它们放入**同一条、固定处于消息
尾部的动态上下文 user message**。字段名只用于宿主更新和排序，不暴露给模型。

尾部消息会明确告诉模型，这些内容是回答上方用户消息所需的最新补充事实，不是新的用户请求。
快照不会进入 system message，也不会插入历史中间。没有快照文本时，不生成快照内容。

适合放入快照的是会变化且应以宿主为真相源的当前业务状态，例如当前文档、订单、选区或页面
对象。稳定角色规则放在 role Skill；任务相关能力规则放在 feature Skill；需要形成对话事实的内容写入 ledger。

## 9.3 摘要的位置

摘要不是快照，也不属于 system message。Runtime 在压缩历史后写入 `Summary` ledger
记录；该记录不直接渲染到前端聊天列表，但在下一轮提供给 LLM 时会被渲染为一条 user
上下文消息：

```text
以下是此前对话的压缩摘要，供你延续上下文：
<summary content>
```

摘要位于 conversation history 中，后面跟随摘要之后保留的历史和当前用户消息；最后才是
动态上下文消息。因此，只要同时存在摘要和快照，模型看到的顺序就是：

```text
system
-> summary and subsequent history
-> current user message
-> final dynamic context containing the latest snapshot
```

摘要保存“此前发生了什么”，尾部快照提供“此刻外部状态是什么”，两者不能合并或互相替代。

## 9.4 使用场景

快照优先承载**变更频率高，并且会显著影响任务判断准确性**的外部状态。典型特征是：模型调用
工具后，工具可能导致前端或外部环境发生不可预测的变化；或者变化虽然可预测，模型仍然必须
根据操作后的真实状态判断任务是否完成以及下一步该做什么。

典型场景包括：

- 浏览器页面结构：点击、跳转、弹窗、异步加载和局部刷新都可能改变可交互元素；
- 前端应用状态：当前视图、选中对象、表单值、加载状态、错误状态和可用操作；
- 编辑器状态：当前文档、选区、光标、诊断结果和工具执行后的文件结构；
- 工具修改业务对象后，模型仍需依据最新状态验证结果或决定后续动作的工作流。

例如，模型调用点击工具后，页面可能完成跳转、打开弹窗、进入加载状态，也可能因校验失败而
保持原状。不能只根据“点击成功”推断任务结果；宿主应更新浏览器结构或前端状态快照，让模型
在下一轮依据操作后的真实界面继续判断。即使某次变化理论上可预测，只要任务完成条件依赖
实际状态，也应重新发布快照，而不是让模型沿用操作前的结构。

因此，快照服务的是一个闭环：

```text
读取当前快照 -> 调用工具改变外部状态 -> 宿主重新采集并覆盖快照 -> 根据最新状态继续或验收
```

低频、稳定，或者不会影响后续工具选择与完成判断的信息，通常没有必要放入快照。

## 9.5 更新方式

快照何时重新采集和注入由宿主决定。Runtime 只提供按 Agent 和字段覆盖快照的能力，并在
thinking 时读取最新值；它不监听宿主业务状态，也不规定统一的刷新策略。

宿主通常可以采用两种方式：

### 9.5.1 监听状态变化

宿主监听自己的前端、浏览器、编辑器或业务状态。当相关状态发生变化时，立即重新采集并调用
`conversation.set_dynamic_snapshot` 覆盖对应字段。这种方式适合状态不仅会被 Agent 工具
改变，也会被用户操作、异步加载、推送事件或其他进程改变的场景。

### 9.5.2 随工具调用更新

宿主也可以把快照更新逻辑写入外部 Tool 的实现中。Tool 完成点击、编辑、跳转或业务修改后，
重新读取操作后的真实状态，再覆盖对应快照，然后才向 Runtime 返回工具执行结果。这样下一轮
thinking 会同时获得工具结果和已经刷新的尾部快照。

```text
Tool 接收调用
-> 执行外部操作
-> 等待本次操作产生的必要状态变化
-> 重新采集真实状态
-> 覆盖快照
-> 返回 Tool 结果
-> 下一轮 thinking 读取最新快照
```

两种方式可以同时使用：状态监听保证用户操作和异步变化不会遗漏，Tool 内更新保证 Agent 发起
的关键操作在返回后已经完成一次有明确时序的重新观测。无论采用哪种方式，同一
`field_name` 都应表达同一种状态含义，并使用覆盖语义，避免并行更新把较旧的观测写回到较新
观测之上。

快照不适合承载：

- Agent 的身份、职责和行为规则，这些属于 role Skill；
- 某类任务的工具使用规则，这些属于 feature Skill；
- 用户和 Agent 已经完成的交流、决定和工具结果，这些属于 ledger；
- 需要跨 Runtime 重建的持久化数据，宿主应在自己的持久化层保存，并在需要时重新发布快照；
- 密钥、访问令牌等传输凭据。

## 9.6 设计理念

快照遵循“宿主掌握当前状态，ledger 掌握对话历史”的边界。模型不负责根据工具调用结果猜测
浏览器、前端或编辑器现在应当是什么样子；宿主在操作后重新采集真实状态，Runtime 保证最新
观测靠近下一轮推理入口。

它采用覆盖而不是追加语义。`field_name` 是稳定槽位，同一字段的新值替换旧值，避免每次状态
变化都向 ledger 追加消息并造成上下文膨胀。多个槽位最终仍合成为一条尾部消息，避免把一次
宿主状态同步伪装成多个用户请求。

它被固定放在消息尾部，是为了同时满足三个目标：

1. **新鲜度**：当前状态不会被后续历史消息覆盖或埋没；
2. **上下文隔离**：动态业务状态不污染 system、Skill 和长期对话历史；
3. **确定性**：无论历史是否被摘要，模型都在相同位置读取本轮最新外部事实。

快照只提供事实，不应写成新的任务指令。真正的用户意图仍来自当前用户消息；快照只是回答该
消息时采用的当前环境。这也是尾部模板明确声明“不是新的用户请求”的原因。
